# v4 — additional providers (deliberately brief)

The provider pattern is the documentation. To add OpenAI/Azure OpenAI (or any other):

1. One new file `src/providers/openai.ts` implementing the `ModelProvider` interface,
   hand-rolled over raw HTTP exactly like `anthropic.ts` (SSE) — read that file first;
   it is the reference implementation. No SDK (locked decision).
2. Register it in the factory (`src/providers/index.ts`) — the ONLY other file that
   may mention it. CLI `--provider openai`; key via `OPENAI_API_KEY` env var
   (Azure additionally needs endpoint + deployment env vars; refuse to start without
   them, with a message naming each missing variable).
3. Offline tests with injected fetch mirroring `tests/providers/anthropic.test.ts`:
   happy path, SSE parse edge cases (split chunks, keep-alives), HTTP error mapping,
   timeout semantics (timeouts never retry; fast network failures do — see
   src/providers/http.ts and copy its usage, not its code).
4. A live smoke test behind `SMOKE_OPENAI=1`, and a scorecard row via one full eval
   run before claiming support in the README.

Failure playbook: streaming bugs are ALWAYS parser-state bugs first — write the
failing chunk sequence as an offline test before touching the parser. Provider
differences in JSON-mode reliability are absorbed by the existing repair loop; do not
add provider-specific prompt hacks.

That is the whole of v4. Anything more elaborate is scope drift.
